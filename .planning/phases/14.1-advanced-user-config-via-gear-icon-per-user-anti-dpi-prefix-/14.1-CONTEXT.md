# Phase 14.1 — Advanced user config via gear icon

## Objective

Расширить Users tab Pro-приложения: добавить полный управление пользователями через **модалку Add/Edit с двумя секциями (credentials + deeplink params)**, включая **per-user anti-DPI client_random_prefix** (rules.toml), **CIDR access restriction** (rules.toml), **password rotation**, и все 7 опциональных полей deeplink TLV spec (anti_dpi, name, custom_sni, upstream_protocol, skip_verification, certificate pinning, dns_upstreams).

Phase 14 завершила базовый редизайн tab'а (список + add/remove + QR export). Phase 14.1 реализует upstream spec'овые фичи которые были пропущены в базе: per-user anti-DPI, CIDR, custom deeplink params. Основан на `memory/users-tab-upstream-audit.md`.

## Scope (locked via discussion 2026-04-18)

### IN scope

- **UserModal** (new component, replaces inline `UsersAddForm`) — Add/Edit unified modal with 2 visible sections:
  - «Учётные данные» (credentials.toml persisted): username + password (+ rotation button in Edit)
  - «Параметры deeplink» (deeplink TLV + rules.toml): all 7 optional fields + CIDR + anti-DPI prefix toggle
- **Plus-icon** в header Users секции (открывает UserModal в Add mode)
- **Gear-icon** per row (3-й icon рядом с FileText+Trash), открывает UserModal в Edit mode
- **CIDRPicker** (new shared/ui primitive) — structured IP/prefix picker
- **Password rotation sub-flow** — кнопка «Сменить пароль» в Edit modal → sub-prompt → atomic backend `server_rotate_user_password`
- **Warning banner** в Edit modal при dirty state в Deeplink секции
- **Backend Rust commands** (new):
  - `server_add_user_advanced` — atomic create с rules.toml (prefix + cidr) + embedded deeplink params
  - `server_update_user_config` — обновить rules.toml per user (CIDR, prefix)
  - `server_rotate_user_password` — atomic remove+add с тем же username
  - `server_regenerate_client_prefix` — regenerate anti-DPI prefix
  - `server_fetch_endpoint_cert` — для certificate pinning UI (returns DER chain + fingerprint)
  - `server_export_config_deeplink_advanced` — replace существующий с support для всех 7 TLV опций

### OUT of scope (deferred)

- **Bulk operations** (multi-select + bulk delete/export) → Phase 14.2 (QoL, backlog)
- **User labels / notes** (localStorage client-side) → Phase 14.2
- **Activity timeline per user** (filter `activity.log`) → Phase 14.2
- **Import from deeplink** (migration flow) → Phase 14.3 (standalone feature)
- **Connection test** («проверить что этот deeplink работает») → backlog
- Server config ETL (vpn.toml settings UI) → Phase 17 Utilities tab territory

## Decisions (D-1..D-9)

### D-1: Modal structure = two visible sections (option B)

Модалка имеет 2 **всегда видимые** секции: «Учётные данные» + «Параметры deeplink» (не collapsible, не tabs). Все поля sub-секции deeplink видны сразу — user не искает «где anti-DPI toggle».

**Implications:** модалка получится высокой (10+ полей). Scroll внутри modal'ного body приемлем. `size="lg"` Modal primitive.

### D-2: Add flow = plus-icon in header (option A)

Существующий `UsersAddForm.tsx` (inline форма под списком) **удаляется**. В header Card'а Users добавляется **plus-icon button** → открывает UserModal в Add mode.

**Implications:**
- `UsersAddForm.tsx` + `UsersAddForm.test.tsx` удаляются
- `UsersSection.tsx` refactor — убрать `<UsersAddForm>` из render, убрать handlers (handleAddUser)
- Pre-fill логика (generateUniqueUsername + generatePassword) переезжает в UserModal mount effect
- i18n ключи от UsersAddForm перевозятся в UserModal namespace

### D-3: Row icons = 3 icons (option B)

Каждый user row получает **3 inline иконки** справа: `FileText` (показать конфиг — открывает UserConfigModal как в Phase 14) + `Gear` (редактировать — открывает UserModal в Edit mode) + `Trash` (удалить).

**Implications:**
- Тесновато при узкой ширине панели — при планировании разметить spacing (icon cluster gap-1 → gap-0.5?) или рассмотреть overflow strategy
- Tooltip на каждом обязательно (уже есть pattern из Phase 14)
- Новый i18n key: `server.users.edit_tooltip`

### D-4: Deeplink params coverage = all 7 TLV fields (option B)

Все 7 опциональных TLV-полей из DEEP_LINK spec показываются в UI:

| Field | Type | UI |
|-------|------|-----|
| `anti_dpi` (0x0A) | boolean | Toggle, default ON (D-5) |
| `name` (0x0C) | string | Text input, max 64 chars |
| `custom_sni` (0x03) | string (FQDN) | Text input + validation |
| `upstream_protocol` (0x09) | enum | Select: «auto» / «HTTP/2» / «HTTP/3» |
| `skip_verification` (0x07) | boolean | Toggle + warning «⚠ dev only, unsafe» |
| `certificate` (0x08) | DER bytes | Toggle «Pin certificate» + [Загрузить из endpoint'а] button → fetch + display fingerprint + embed |
| `dns_upstreams` (0x0D) | string[] | Multi-line input (one DNS per line) + validation IP/hostname |

### D-4 Footnote (added 2026-04-18 per Plan 14.1-00)

**Upstream CLI gap resolved:** See [memory/users-tab-upstream-audit-phase14.1.md](../../../memory/users-tab-upstream-audit-phase14.1.md). Path A chosen — all 7 fields shown. Fields `anti_dpi` (0x0A), `skip_verification` (0x07), `upstream_protocol` (0x09), `certificate` (0x08) will be added client-side in Rust via TLV post-encoding (Plan 14.1-01 `tlv_encoder.rs` sub-module, round-trip tests against upstream `deeplink/src/encode.rs` fixtures).

### D-5: Anti-DPI default = ON (option A)

При Add нового user'a toggle `anti_dpi` включён по умолчанию. Клиент автоматически запрашивает backend с `--generate-client-random-prefix --prefix-length 4 --prefix-percent 70`. Пользователь может выключить вручную (для тест/debug сценариев).

**Implications:**
- `server_add_user_advanced` должен генерить prefix + записывать в rules.toml + embed'ить в deeplink за один atomic вызов
- Если `anti_dpi` OFF — ни rules.toml entry, ни TLV field, basic deeplink

### D-6: Certificate pinning = in scope (via D-4)

Опция D-4 «все 7 полей» автоматически включает certificate pinning. Требует:
- Backend `server_fetch_endpoint_cert(sshParams)` — вызывает `openssl s_client` или native TLS probe на целевой endpoint, возвращает DER chain + SHA-256 fingerprint
- UI: Toggle «Закрепить сертификат endpoint'а» → при toggle-on → fetch → показать fingerprint в мини-карточке → embed в deeplink

### D-7: Password rotation = separate button (option A)

В Edit modal поле Password **read-only** (показано как `•••••`). Рядом кнопка «Сменить пароль» → sub-prompt (или отдельная мини-модалка) с полем «Новый пароль» (+ shuffle + validation как в Phase 14) → Confirm → backend `server_rotate_user_password(username, new_password)` atomic.

**Implications:**
- Backend должен гарантировать atomicity: remove old user + add new с тем же username (либо native «update password» если сервер поддержит в CLI — research task)
- Warning обязательно: «Старые deeplink'и пользователя перестанут работать»
- Activity log: `user.password.rotated user=X`
- i18n ключи: `server.users.rotate_password`, `server.users.rotate_password_confirm_title`, `server.users.rotate_password_warning`

### D-8: CIDR input = structured picker (option B)

Отдельный shared/ui primitive **`CIDRPicker`** с разметкой:
```
IP:     [___] . [___] . [___] . [___]
Prefix: [__ ▼]  (select 0-32)
```
+ inline preview рассчитанного диапазона («192.168.0.0/24 — 256 addresses»).

**Implications:**
- Новый shared/ui компонент с unit-тестами + Storybook stories
- Validation: octets 0-255, prefix 0-32, «0.0.0.0/0» = «без ограничений»
- Пустое состояние = без CIDR restriction (deployed rule не содержит `cidr=` ключ)

### D-9: Regenerate deeplink warning = banner on dirty (option A)

Warning banner в top of Edit-модалки показывается **когда user изменил любое поле из секции «Параметры deeplink»** (dirty tracking на эту секцию):

```
┌────────────────────────────────────────────┐
│ ⚠ После сохранения старые deeplink'и этого │
│   пользователя перестанут работать         │
└────────────────────────────────────────────┘
```

Неблокирующий banner, info/warning severity. Не confirm dialog (aggressive), не inline note (недостаточно visible).

**Implications:**
- Dirty tracking на section level (compare initial deeplink params with current form state)
- Existing `ErrorBanner` primitive может работать с `severity="warning"` — проверить в research
- Banner hidden когда user меняет только секцию credentials (смена пароля не ломает deeplink)

## Architectural notes

### Backend work estimate

Phase 14.1 требует ~5-6 новых Rust commands. Это **значительная backend работа**, ~3-5 дней отдельно (parse deeplink TLV, atomic rules.toml editing, openssl TLS probe, etc.). Во время планирования выделить в отдельный plan «Backend» (14.1-01 или 14.1-02).

### Component new list

1. **UserModal** (replaces UsersAddForm) — main Add/Edit modal
2. **CIDRPicker** (new shared/ui primitive) — IP+prefix picker с validation
3. **PasswordRotationPrompt** (sub-component inside UserModal) — смена пароля flow
4. **CertificateFingerprintCard** (sub-component inside UserModal «Параметры deeplink» секции) — fetch + display cert
5. Возможно **DnsUpstreamsInput** (sub-component) — multi-line DNS list input

### Deeplink generation flow

Все deeplink-параметры (D-4 all 7 fields) запекаются **в момент генерации** deeplink'а через `server_export_config_deeplink_advanced(username, { anti_dpi, name, custom_sni, ... })`. Сервер НЕ хранит эти настройки отдельно — они живут только в выданном tt://.

**Следствие для Edit modal:**
- При открытии Edit — поля «Параметры deeplink» инициализируются **default values** (не retrieved from server — он их не хранит)
- User может видеть «текущий выданный deeplink» (если сохранён в clipboard history / localStorage?) но это out of scope
- «Regenerate deeplink» в Edit = generate new с новыми params, которые user задал в этой сессии

### What server DOES store per user

- `credentials.toml` — username + password
- `rules.toml` (если anti-DPI или CIDR заданы) — [[rule]] с `client_random_prefix` + `cidr`

Edit modal читает **только** эти данные через new command `server_get_user_config(username)`. Всё остальное (deeplink params) — session-scoped form state.

## Success criteria (preview, planner detalises)

- UserModal Add: плюс-icon → модалка с 2 секциями → Add заполняет credentials.toml + (опц) rules.toml + генерит deeplink с всеми TLV fields выбранными user'ом → auto-open UserConfigModal с новым QR

- CIDRPicker: IP+prefix input с validation (octets 0-255, prefix 0-32), empty = «без CIDR rule»
- Password rotation: отдельная кнопка в Edit → sub-prompt → atomic backend command → old deeplink'и инвалидируются, SnackBar confirm
- Certificate pinning: toggle → fetch endpoint cert → fingerprint displayed → embed in deeplink → Light-client validates
- Anti-DPI ON by default → каждый новый user получает уникальный `client_random_prefix` + rule в rules.toml
- Row layout: 3 icons (FileText+Gear+Trash) не overflow'ят при minWidth 800 window
- Full test coverage: UserModal + CIDRPicker + backend commands Rust tests + password rotation flow integration
