### WR-01: Fire-and-forget Promise pattern hides real SSH write failures from the user

**File:** `gui-pro/src/components/server/UsersSection.tsx:266-276`
**File:** `gui-pro/src/components/server/UserModal.tsx:796-824`, `1008-1036`

**Issue:**
Several call-sites wrap `invoke(...)` in `Promise.resolve(...).catch(...)` with a comment about test mocks returning non-Promise values. Production `invoke` always returns a Promise — the wrapper is unit-test driven. Functional consequence: backend SSH writes (e.g. `server_set_user_advanced`, `server_delete_user_advanced`) are fire-and-forget. If the network drops between the success-toast call and the backend write, the operator sees "saved" but `users-advanced.toml` never updated, and the inconsistency surfaces only on the next Edit reopen.

This is worse for `server_delete_user_advanced` after `server_remove_user` — credentials.toml is gone but advanced-file has orphan entry. Not catastrophic (the next `server_reconcile_users_advanced` call cleans it up) but means D-29 "operator always knows what happened" invariant wobbles.

**Fix:**
```ts
// Remove Promise.resolve wrapper. If tests break, fix the mock:
//   vi.mocked(invoke).mockResolvedValue(undefined);
void invoke("server_set_user_advanced", { ... }).catch((err) => {
  activityLog("ERROR", `user.advanced.persist_failed user=${username} err=${formatError(err)}`);
  // Optionally surface via toast: pushSuccess(t("server.users.sync_background_failed"), "warning");
});
```

Or — if "best-effort" is the deliberate design — document in `UsersSection.tsx` near `handleUserAdded` that the `server_set_user_advanced` call is a safety net only (backend's `server_add_user_advanced` Step 4 already performs the write), so callers don't mistake failure for data loss.

### WR-02: Stale closure risk in `UsersSection` tab-activation effect vs SSH-port-change flow

**File:** `gui-pro/src/components/server/UsersSection.tsx:122-156`

**Issue:**
The M-04 tab-activation effect depends only on `activeServerTab`:

```tsx
useEffect(() => {
  if (activeServerTab !== "users") return;
  // ... state.loadServerInfo(silent) + refreshDisplayNames + server_reconcile_users_advanced ...
}, [activeServerTab]);   // eslint-disable-next-line react-hooks/exhaustive-deps
```

Inside the body it references `state.loadServerInfo`, `refreshDisplayNames` (which captures `sshParams`), `sshParams`, `activityLog`. Scenario that breaks:

1. Operator opens Users tab → effect fires with `sshParams{host, port=22}`.
2. Security-tab flow calls `security_change_ssh_port` → pool invalidates → `sshParams` updated to `port=2200`.
3. `activeServerTab` stayed `"users"` — effect NEVER re-fires. The `refreshDisplayNames` callback captured at mount still holds the stale params.
4. On the NEXT tab-activation trigger (deactivate → reactivate), fresh callback runs — but the ~30s window in between leaves stale-pooled requests.

`OverviewSection.tsx:262-312` has the right pattern (ref-based stable handle `rebootRefs.current` updated on every render); `UsersSection` should mirror it.

Related: `refreshDisplayNames` cannot be cancelled mid-flight. If the user double-clicks the Users tab, two parallel SSH reads race → `setDisplayNames` called twice, last-write-wins (benign here, but smells).

**Fix:**
```tsx
const tabRefs = useRef({ sshParams, loadServerInfo: state.loadServerInfo, refreshDisplayNames, activityLog });
tabRefs.current = { sshParams, loadServerInfo: state.loadServerInfo, refreshDisplayNames, activityLog };

useEffect(() => {
  if (activeServerTab !== "users") return;
  if (firstTabActivationRef.current) { firstTabActivationRef.current = false; return; }
  const { sshParams, loadServerInfo, refreshDisplayNames, activityLog } = tabRefs.current;
  // ... body uses refs ...
}, [activeServerTab]);
```

Or add a cancellation ref (pattern in `UserModal.tsx:413`): `let cancelled = false; return () => { cancelled = true; };`, guard every setState with `if (cancelled) return`.

### WR-03: `server_regenerate_client_prefix` silently drops existing CIDR (security regression)

**File:** `gui-pro/src-tauri/src/ssh/server/server_install.rs:633-652`

**Issue:**

```rust
pub async fn server_regenerate_client_prefix(
    app: &tauri::AppHandle,
    handle: &client::Handle<SshHandler>,
    vpn_username: String,
    prefix_length: u32,
    prefix_percent: u32,
) -> Result<String, String> {
    // ...
    // Delegate update with regenerate_prefix=true; cidr preserved from existing rule
    server_update_user_config(app, handle, vpn_username, None, true, true).await?;
    //                                                     ^^^^ — !
    Ok(new_prefix)
}
```

The comment says "cidr preserved from existing rule" — this is false. `server_update_user_config` (`:603-608`) does:

```rust
let removed = super::remove_user_rule(&content, &username)?;
let updated = super::add_user_rule(&removed, &username, new_prefix.as_deref(), cidr.as_deref())?;
```

When the caller passes `cidr=None`, `add_user_rule` writes the new allow rule **without** cidr — so a user who previously had `{prefix="aabb", cidr="10.0.0.0/24"}` ends up with `{prefix="ccdd"}` **no CIDR**. They can now connect from any subnet. The admin invoked "regenerate prefix" (a rotation operation) and got "regenerate prefix + drop subnet restriction" silently.

The comment is lying; either the code or the comment is wrong. Based on user intent of "regenerate" (rotation of a single value), the comment is right and the code needs fixing.

Secondary concern: this path is currently unreachable from the UI (UserModal only calls `server_update_user_config`, not `server_regenerate_client_prefix`), but the command is registered in `lib.rs:354`. If a future UI wires it up, regression lands silently.

**Fix:**
```rust
pub async fn server_regenerate_client_prefix(
    app: &tauri::AppHandle,
    handle: &client::Handle<SshHandler>,
    vpn_username: String,
    prefix_length: u32,
    prefix_percent: u32,
) -> Result<String, String> {
    validate_vpn_username(&vpn_username)?;
    // ... existing prefix generation ...

    // Preserve existing CIDR before delegating (security invariant).
    let sudo = detect_sudo(handle, app).await;
    let (content, _) = exec_command(
        handle, app,
        &format!("{sudo}cat {dir}/rules.toml 2>/dev/null || echo ''", dir = ENDPOINT_DIR),
    ).await?;
    let existing_cidr = super::find_user_rule(&content, &vpn_username)
        .ok()
        .flatten()
        .and_then(|r| r.cidr);

    server_update_user_config(app, handle, vpn_username, existing_cidr, true, true).await?;
    Ok(new_prefix)
}
```

Alternative: document behaviour ("WARNING: resets cidr to None") and add a separate `server_regenerate_client_prefix_keeping_cidr` that reads existing + re-applies. Pick one.

### WR-04: systemctl restart blast radius — operator UX leaves active clients to be disconnected silently

**File:** `gui-pro/src-tauri/src/ssh/server/server_install.rs:225-229`, `282-284`, `365-368`, `395-396`, `623-627`

**Issue:**
Every config-mutation handler ends with:
```rust
let _ = exec_command(handle, app, &format!("{sudo}systemctl --no-block restart trusttunnel")).await;
```

This tears down the upstream `trusttunnel_endpoint` process — and with it every active VPN session on the server. In `server_update_user_config` (`:619-627`) the Russian code comment even says "CIDR / anti-DPI правила читаются upstream'ом только при старте (`RulesEngine::from_config`) — не hot-reload. Без рестарта изменённые `rules.toml` никогда не применяются" — so the restart is **correct and necessary** for `rules.toml` changes.

But it happens silently: operator in Pro UI does "add user with CIDR" and every other connected client on that server gets disconnected. Worse, `server_add_user_advanced` restarts TWICE when CIDR is set — once in `add_server_user` (`:226-229`) inside Step 1, and again in `server_update_user_config` (`:624-627`) inside Step 2. That's two disconnect-reconnect cycles for what looks like a single admin action.

Not a correctness bug, but a UX/security concern for multi-tenant deployments. D-29 "operator always knows what happened" is violated for _other users'_ clients.

**Fix:**

1. Coalesce restarts in `server_add_user_advanced`: suppress the restart in `add_server_user` when called from the advanced wrapper, do a single restart at the end of Step 2 (or explicitly in Step 4).
2. UI surface: show an info-banner on "Add user with CIDR" / "Update user config" confirming "saving will briefly disconnect other active clients (~2-5 s)". Translations key pattern `server.users.warn_restart_impact`.
3. Investigate upstream CLI's support for `SIGHUP` / `systemctl reload`. If added, swap `restart` → `reload` for rules-only changes.

Priority warning, not critical. Behaviour is correct, just not operator-friendly.

### WR-05: CIDR paste handler under-validates octet count + silent drops beyond 4

**File:** `gui-pro/src/shared/ui/CIDRPicker.tsx:141-179`

**Issue:**
`tryFanOutPaste`:
```ts
const nums = body.match(/\d{1,3}/g);
if (!nums || nums.length < 4) return false;
const first4 = nums.slice(0, 4);
```

Several under-documented edge cases:

- Paste `10.20.30.40.50` → `nums = ["10","20","30","40","50"]`, first4 accepted silently. User may have meant a 5-octet typo and got a valid 4-octet IP without feedback.
- Paste `xxx10xxx20xxx30xxx40/24` → 4 numeric matches, still a "valid" CIDR. Harmless, but inconsistent with "structural" paste intent.
- Paste `10.0.0.0/3a4` → `prefixPart.match(/\d{1,2}/)` = `"3"` → prefix becomes 3 (intended was 3? or user typo?). Silent truncation.
- `setOctetErrors([false,false,false,false])` (line 174) correctly clears stale errors — OK.

The `onPaste` handler dispatches to every octet input, which means `Ctrl+V` on octet#3 of a partial CIDR also triggers the fan-out. Correct by design but can be surprising (user meant "paste into this field only").

**Fix:**

1. Add `if (nums.length > 4) return false;` to skip ambiguous pastes — lets the default paste behaviour handle it and makes the user notice.
2. Add unit tests for edge cases (>4 octets, malformed prefix, leading zeros). Current `CIDRPicker.test.tsx` already covers happy paths; deep-pass adds these.
3. Consider surfacing a brief snack-bar when paste was "fanned out" vs "default to single field" — improves discoverability.

### IN-06: `userAdvanced.ts:153-156` — `isLikelyCaChain` uses base64 char count where docstring describes decoded byte count

**File:** `gui-pro/src/shared/utils/userAdvanced.ts:153-156`

**Issue:**
```ts
export function isLikelyCaChain(certDerB64: string | null): boolean {
  if (!certDerB64) return false;
  return certDerB64.length > 2500;
}
```

Docstring says `"larger than ~2.5 KB is almost certainly a CA-issued chain"` but the implementation checks base64 LENGTH. Base64 encoding has ~33% overhead (3 bytes → 4 chars), so:
- 2500 base64 chars ≈ 1875 decoded bytes — not 2.5 KB.
- A 4096-bit self-signed leaf with several extensions is easily 1.5-2 KB decoded ≈ 2000-2700 base64 chars → could trip the threshold → `is_system_verifiable = true` → cert gets **stripped** from the deeplink → handshake fails.

Edge case is rare but possible. The real fix is to persist `certIsSystemVerifiable` server-side (currently absent from `UserAdvanced` struct). Heuristic is inherently wrong for the boundary case.

**Fix (minimal):**
```ts
export function isLikelyCaChain(certDerB64: string | null): boolean {
  if (!certDerB64) return false;
  // 2.5 KB decoded ≈ 3333 base64 chars (accounting for ~33% overhead).
  return certDerB64.length > 3333;
}
```

**Fix (correct):** Add `pin_cert_is_system_verifiable: bool` to `UserAdvanced` (users_advanced.rs:50-71). Persist explicitly, remove the heuristic on the TS side. Requires file-format migration — consider a later phase.

## Cross-file integrity check (deep-depth findings)

### Pool command registration ↔ handler mapping

### CIDR allow+deny invariant (CONFIGURATION.md §Rules first-match + fall-through=allow)

`server_rules.rs::add_user_rule` (lines 42-85) correctly emits:
- `cidr + prefix` → allow with cidr + deny with prefix only (actual IP restriction works).
- `cidr` without `prefix` → allow with cidr only, **CIDR does NOT block** (fall-through allow). The comment (`:38-41`) explains this and recommends frontend force `anti_dpi=true` when CIDR is set.

### UserModal + CertificateFingerprintCard + userAdvanced — certificate pinning flow

**Add mode (new user):**
1. User toggles Pin Cert ON → CertificateFingerprintCard probes → returns `{leaf_der_b64, fingerprint_hex, is_system_verifiable}`.
2. UserModal stores into `deeplink.certDerB64/certFingerprint/certIsSystemVerifiable`.
3. On Submit: `handleAdd` passes `pinCertificateDer: deeplink.pinCert && !deeplink.certIsSystemVerifiable ? certDerB64 : null` — strips the cert when platform verifier already trusts it (FIX-OO-7 — avoids QR capacity overflow).
4. Backend `server_add_user_advanced` decodes base64 → raw bytes → passes to tlv_encoder which caps at 32 KB.
5. `server_set_user_advanced` persists full `certDerB64` (no strip in storage — FIX-OO-12) so Edit reopen sees the toggle ON.

**Edit mode (reopen):**
1. `Promise.all([server_get_user_config, server_get_user_advanced])` loads two parallel files.
2. `advancedFromServer` converts snake_case → camelCase.
3. `isLikelyCaChain(advanced.certDerB64)` restores the `is_system_verifiable` flag via heuristic (see IN-06 for concern).
4. `deriveFingerprintFromDerB64(certDerB64)` recomputes SHA-256 locally — UserAdvanced doesn't persist the hash.
5. CertificateFingerprintCard hydrates success-state via `initialFingerprint + initialDerB64 + initialIsSystemVerifiable`.

Contract is self-consistent. `isLikelyCaChain` is the weak link (IN-06).
